# Ревью класса Manager

## Синтаксические ошибки

- `_init__` вместо `__init__`
- `def` написан после имени метода, а не перед
- Пропущены `=` в присваиваниях: `self.d data`, `self.time datetime.now()`, `self.status 2` и т.д.
- `z 1` и `z=2` вместо `z == 1` и `z == 2`
- `r.status code` — пробел вместо `_`
- `mode "B"` вместо `mode == "B"`
- `CONFIG {"debug": True}` — нет `=`
- Пустое тело у `if self.d[i] == 0:` — нет оператора

## Проблемы архитектуры

- Метод `do` делает слишком много: обновляет данные, пишет в файл, делает HTTP-запрос, меняет статус и применяет бизнес-логику. Нужно разбить на отдельные методы.
- Название `do` не говорит что метод делает — нужно понятное имя.
- Магические числа 1, 2, 3, 99 для статусов — лучше использовать `Enum`.
- `CONFIG` — глобальный изменяемый словарь. Лучше передавать через `__init__`.

## Работа с ресурсами и ошибки

- `open()` без `with` — файл не закроется при исключении.
- Голый `except:` ловит всё, включая `KeyboardInterrupt`. Нужно писать конкретный тип, например `except requests.RequestException`.

## Кодстайл

- Нет type hints и docstring.
- `type(self.d) is not list` — лучше `not isinstance(self.d, list)`.
- Короткие имена `d`, `s`, `x`, `y`, `z`, `t` ничего не говорят о содержимом.
- `print` для отладки — лучше использовать `logging`.
- Хардкоженный URL в `__init__` — лучше передавать снаружи.
- Двойной цикл для обнуления ненулей можно заменить одним: `[0 if v != 0 else v for v in self.d]`.
- `if mode == "A" or mode "B" or mode == "C"` → `if mode in ("A", "B", "C")`.
